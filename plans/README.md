## Effect.TS / framework-migration question — considered and rejected

A full migration to Effect.TS (or another effect-system framework) was evaluated and
**rejected**. Rationale: the codebase's actual pain points — missing security boundaries,
9000-line god files, and scattered check-then-act races — are not what an effect system
fixes. Migration is viral (any `Effect<A,E,R>` return type propagates through the call
chain) and would require rewriting ~2100 source files deeply integrated with Elysia,
Drizzle, and three AI-provider SDKs — multi-person-month work with a functionality freeze
and a new team-wide mental model, introducing new bugs while fixing none of the above.
The equivalent capabilities already exist in lighter form (`app.onStop` for resource
cleanup, `AppError` + error-mapping for errors, native Promises for concurrency); what's
missing is discipline on existing patterns, delivered by plans 010–012. If typed errors
are desired, adopt `neverthrow` locally in the provider layer rather than Effect globally.
A scoped Effect.TS experiment inside a _new, isolated_ subsystem (e.g. a future relay
orchestration layer) would be reasonable; a whole-codebase migration is negative-value.
