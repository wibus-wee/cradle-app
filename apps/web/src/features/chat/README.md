# Features/Chat

Renderer-side view layer for chat.
Streaming updates arrive as AI SDK `UIMessageChunk` frames through a chat-owned transport abstraction. Ordinary web runtime uses server SSE directly. Electron Desktop runtime uses `window.cradle.chatStream`, where Electron main owns the long-lived upstream server SSE connection and renderer windows consume IPC fanout. Initial hydration and reload/recovery read trusted `UIMessage` snapshots through React Query and project locally into Zustand. Terminal snapshot rows are also mirrored into a Cradle-owned IndexedDB cache so remounting a previously opened session can restore stable transcript content before the canonical server snapshot request returns; streaming rows never enter that cache.
Passive windows hydrate the current message snapshot first, then subscribe through `chat-stream-transport.ts` for active-run AI SDK chunks instead of polling snapshots while a message is streaming. Web runtime uses the multiplex `/sync` WebSocket when enabled and otherwise calls `GET /chat/sessions/{sessionId}/stream`; Electron Desktop subscribes to the main-process broker so multiple windows share one upstream stream. The WebSocket client owns ping/pong deadlines, socket generations, reconnect backoff, and exactly-once subscription registration. Run chunks resume only from the last accepted `{ runId, cursor }`; backpressure/upstream interruption reconnects, while `snapshot-required` fails the passive stream into its existing canonical snapshot refresh instead of reporting normal EOF. The passive chunk consumer always rebuilds the active assistant message from the replayed stream protocol instead of using the hydrated snapshot as AI SDK internal stream state, so late joins receive required part-start chunks before text/reasoning deltas even when the DB snapshot is still empty during streaming.
Codex active goals can create server-owned system continuation runs after a previous run fails or idles without user cancellation. Chat Runtime event-tail invalidation refreshes `/runtime-status` and message snapshots when run or interaction facts change, and `use-chat-session.ts` refreshes message snapshots when runtime status exposes an active run message id that is not yet hydrated. Once the new streaming assistant row appears in the snapshot, the existing passive session-stream join path consumes the new run; no visible internal user message is added to chat history.
Text/reasoning/file parts hydrate as `UIMessage.parts`. Chat-owned structured context parts such as `$`-selected Skills are also stored in `UIMessage.parts` under the `data-cradle-*` namespace, rendered inline with the user prompt, and projected only by providers that understand them. Tool calls also stay in `UIMessage.parts`; chat render plans derive transient `messageId + partIndex` refs for subscriptions, but no browser-owned mirror stores tool payload by `toolCallId`.
The store preserves raw AI SDK `UIMessage.parts` as the message-history source of truth. Tool UI reads fields such as `approval`, `input`, `providerExecuted`, output state, and provider-specific payload directly from the current tool part so native AI SDK approval continuation and Cradle Tool UI standardization evolve on the same message boundary.
Native AI SDK tool approvals render inline from tool parts with `state: "approval-requested"`. Approve/deny actions update the tool part to `approval-responded`, use AI SDK `lastAssistantMessageIsCompleteWithApprovalResponses`, and continue by sending the complete `messages` history through the existing chat response endpoint. Runtime-owned approval requests use the same inline buttons but submit to Chat Runtime's pending tool-approval endpoint so the active provider request can resume. Codex app-server approvals are identified by standardized `approval.*` tool payloads; Claude Agent `canUseTool` approvals are identified by the Claude Code builtin tool envelope. Neither path uses the deleted independent approval stream.
Claude Agent `TodoWrite` state is synthesized by the server adapter into persisted tool output `pluginState.todos`. Chat store selectors reverse-scan raw `UIMessage.parts` for the latest plugin state and render it in Right Aside runtime surfaces; inline `TodoWrite` blocks may still project directly from tool args while the call is running. The TODO shared vocabulary is `todo` / `processing` / `completed`, with processing text read from `activeForm`. Codex app-server plan/file/MCP/web tool items render through the same classified tool block surface. Chat Runtime access and interaction controls are Cradle-owned `runtimeSettings`: the Composer control persists session defaults through `/runtime-settings`, sends loaded defaults with new turns and queue items, and lets active providers apply live updates through their runtime side-channel.
Provider-owned UI slot state is consumed by the surface declared by the provider. Composer-adjacent persistent rendering is reserved for slots with `composerState`, such as Codex goal, Codex plan-ready actions, and Codex background terminal controls; ChatGPT account usage is a slash-triggered composer rail opened by `/usage` and closed by the user, while its unavailable state remains visible as a disabled slash command row when no provider state exists. Snapshot-style runtime metadata such as status, model, reasoning, MCP, alerts, skills, plugin, and config must not render as chips above the input or as synthetic message-bubble content. Message-stream events that need transcript placement should arrive as stream evidence or persisted message/tool parts instead of polled UI slot state. Codex compact state separates context-compaction lifecycle from thread token-usage meter state, so the `/compact` row can show running/completed lifecycle and current usage as an icon-ring state plus inline copy like `(used 52%)` without Cradle owning compaction semantics. Picker-style state such as model, reasoning, and config stays in existing model-picker, runtime-panel, or diagnostics surfaces, not by fake `/model` or `/reasoning` slash commands. Right Aside Runtime renders Cradle-owned workflow summaries such as Plan/TODO progress, run status, tools, and chat attention; Dev-only runtime environment diagnostics render from a top-right resource-style popover.
`/btw` is provider-owned quick-question UI exposed through a runtime `quick-question` slot with `slashCommand` and `composerState` surfaces. The slash picker renders it from runtime capabilities, and submitting `/btw <question>` opens the composer-adjacent quick-question slot rail only when that provider slot exists. It streams through the generated `/chat/sessions/{sessionId}/quick-question` SSE SDK entry while the slot renderer validates AI SDK chunks with `uiMessageChunkSchema` and projects text deltas locally. It is not registered as a Cradle-owned fallback command, does not persist a chat message, and does not open a Dialog or Browser Panel.
`/side` is a Cradle-owned chat command, not a provider command. The composer exposes it alongside other Cradle commands, `use-chat-session.ts` intercepts `/side [message]`, calls `POST /chat/sessions/{sessionId}/side-chat`, opens a Browser Panel `side-conversation` tab in the current BrowserPanel owner scope, and optionally submits the inline message/files/context through `POST /chat/side-conversations/{sideConversationId}/response`. Side messages use the renderer-only live chat store under keys such as `side:{id}` and do not create Cradle Chat Session rows.
Subagent detail tabs render provider-native threads through Chat Runtime provider-thread APIs. Codex and Claude Agent subagents hydrate with `provider-threads/:threadId/turns` and subscribe to `provider-threads/:threadId/stream`, then reuse `ChatStreamingHandler`, the renderer-only live chat store, and `MessageBubble` with a synthetic provider-thread view id. This keeps subagent detail on the same AI SDK message/chunk architecture as Chat Session without creating Cradle child sessions, polling UI slot state, scraping the parent tool output, or mixing provider-thread transcripts into the main session store. Parent chat tool rows render only the provider-owned tool lifecycle, status, summary, and a Browser Panel handoff; full subagent transcripts are not nested into parent message parts.
When a session is already streaming, the composer chooses between two distinct follow-up transports. `queue` submits a durable Chat Session continuation through `/chat/sessions/{sessionId}/queue`; those rows are shown in the queue controls and can be cancelled or reordered while pending. `steer` submits live same-turn guidance through `/chat/sessions/{sessionId}/steer`; accepted steer messages are inserted into the visible transcript from the route response and are not queue items. The default mode comes from Chat settings, and `Shift+Meta/Ctrl+Enter` flips between `queue` and `steer` for one send.
AI SDK wire validation is intentionally limited to the SSE boundary through `uiMessageChunkSchema`. Once a frame has been accepted as an AI SDK chunk or a snapshot has been loaded from Cradle's own database, chat feature code treats the object as trusted and uses direct TypeScript narrowing instead of feature-owned Zod schemas.

## Files

- **chat-tool-entities.ts**: Trusted tool-part helpers for reading AI SDK tool names, approval metadata, and builtin tool payload identity from `UIMessage.parts` without materializing a parallel tool entity store
- **chat-capabilities.ts**: Feature-owned GET boundary for `/chat/draft-runtime-capabilities`, `/chat/sessions/{sessionId}/capabilities`, `/chat/sessions/{sessionId}/ui-slot-states`, and background-terminal list/terminate actions, used only for runtime-native capability data and provider-owned slot state such as Codex compact, plan, tool activity, MCP, diff, terminal, approval, alert, filesystem, skills, plugin, search, crew, usage with account rate-limit windows, and config summaries. The response is a Cradle-owned trusted API payload.
- **provider-thread-command.ts**: Chat Runtime provider-thread HTTP boundary for provider-native subagent/thread detail panels, including metadata read, turn hydration with projected `UIMessage` records, and AI SDK chunk stream subscription.
- **chat-context.ts**: Chat-owned attention snapshot store and renderer context provider factory; publishes active chat attention state such as viewport range, scroll position bucket, near-bottom/manual-scroll state, and composer focus without exposing Chat DOM structure to system-agent, while app composition owns provider installation.
- **chat-context.test.ts**: Unit coverage for Chat attention snapshot publication into typed Jarvis context envelopes
- **bang-command.ts**: Chat-owned parser for Composer `!cmd` submissions; accepts only single-line commands with no space after `!`, leaving empty, spaced, or multi-line input as ordinary chat text.
- **bang-command-metadata.ts**: Trusted metadata reader for Cradle-owned `metadata.cradle.bangResult` snapshots persisted by Chat Runtime.
- **chat-slash-commands.ts**: Web-owned slash command descriptor and merge helpers; converts runtime-native commands and command-like runtime UI slots into composer actions, annotates matching runtime slot command rows with provider-owned state labels, filters picker/metadata slots out of the slash panel, executes provider-declared slot `commandAction` metadata for submit/UI actions, hides `requiresSession` slots from pre-session draft composers, registers Cradle UI commands such as `/appshot` and `/side`, and keeps Cradle UI commands separate from provider capabilities
- **message-bubble.tsx**: Store-backed and direct message bubble renderer with execution folding, tool approval actions, copy controls, Codex goal message presentation, bang command result presentation, optional per-surface text display transforms, and tool blocks subscribed by `messageId + partIndex`. It renders persisted message content and stream-derived tool evidence directly from `UIMessage.parts`, but does not inject polled runtime UI slot snapshots into assistant bubbles.
- **optimistic-chat-turn.ts**: Chat-owned optimistic first-turn boundary used by launch surfaces; appends the user prompt and pending assistant bubble immediately, starts the shared chat stream transport, and leaves canonical snapshot reconciliation to the normal chat session refresh path.
- **chat-response-command.ts**: Feature-owned HTTP command boundary for `/chat/sessions/{sessionId}/response`, `/chat/sessions/{sessionId}/bang-command`, live side conversation create/response/delete routes, `/chat/sessions/{sessionId}/stream`, `/chat/sessions/{sessionId}/cancel`, `/chat/sessions/{sessionId}/queue`, and `/chat/sessions/{sessionId}/steer`; long-lived response/stream calls are now consumed through `chat-stream-transport.ts` so Electron can replace renderer-owned SSE with desktop IPC fanout while web runtime keeps the same HTTP path.
