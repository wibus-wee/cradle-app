# Chat Runtime Module

Provides server-owned chat turn execution for existing sessions, including text/file user input, durable continuation queue items, AI SDK `UIMessageChunk` SSE streaming, final `UIMessage` snapshot persistence through AI SDK stream finish callbacks, usage writes, run state updates, cancellation terminal transitions, and ACP live runtime orchestration.
`messages.message_json` is the only hydration source of truth. Stored message snapshots are Cradle-internal trusted `UIMessage` JSON; this module parses them directly instead of running a second Zod projection schema over AI SDK-owned message state.
Provider turn context is intentionally reconstructed through a Cradle-owned transcript layer: ordinary new turns read recent complete top-level messages from `messages.message_json`, preserve the stored `UIMessage.parts` structure, and expose a bounded `CradleTurnTranscript` for provider-specific native history projectors. Native approval continuation still uses the explicit AI SDK `messages` request body because it must preserve tool lifecycle state exactly as supplied by the client. Final assistant snapshots are projected incrementally while chunks are published, so terminal finalization does not replay the active run buffer through AI SDK stream reconstruction. Final snapshots and oversized historical snapshots are compacted with server-owned limits so old tool/reasoning payloads do not create unbounded parse, stringify, replay, or response costs.
`POST /chat/sessions/:sessionId/response` accepts ordinary text/file input, chat-owned `contextParts`, optional model/provider overrides, `low` / `medium` / `high` / `xhigh` thinking effort, and AI SDK `messages` history. The `messages` form is used for native tool approval continuation: the last assistant `UIMessage` already contains the `approval-responded` tool part, and the server reuses that assistant message id so AI SDK can continue the same tool lifecycle through `convertToModelMessages` and `collectToolApprovals`.
Provider plan implementation is exposed through synthetic `plan_implementation` approval parts. `POST /chat/sessions/:sessionId/messages/:messageId/plan-implementation-approval` only persists the local approval response; the renderer submits approved plans as an ordinary follow-up turn using `PLEASE IMPLEMENT THIS PLAN:`.
`POST /chat/sessions/:sessionId/user-input/:requestId` resolves a Chat Runtime-owned pending user input request for the active provider turn. Provider adapters map native request shapes such as Codex `item/tool/requestUserInput` and Claude Agent `AskUserQuestion` into runtime questions, then wait on the runtime promise; submitted answers resolve that promise and publish a synthetic `tool-output-available` chunk so renderers clear the pending prompt after Cradle accepts the submission. While a request is pending for the active run, `/chat/sessions/:sessionId/runtime-status` reports `waitingForUserInput` instead of ordinary `streaming` so status surfaces can distinguish a blocked question from generation. `POST /chat/sessions/:sessionId/tool-approval/:requestId` resolves a separate pending tool approval promise for provider-native requests such as Codex command, file-change, permission, `applyPatchApproval`, `execCommandApproval`, and Claude Agent SDK `canUseTool`; Chat Runtime owns the pending lifecycle and provider adapters own the native response shape. If the provider approval belongs to a provider-native subagent thread, the approval tool chunks are published to that provider-thread stream rather than the parent chat stream.
`POST /chat/sessions/:sessionId/bang-command` executes a user-entered single-line shell command through the session runtime and persists the command plus result as two complete user messages for renderer hydration. Standard runtimes use Chat Runtime's local executor from the session workspace. Codex runtime delegates execution to the Codex provider, which calls Codex app-server `thread/shellCommand` so Codex owns the native `commandExecution` protocol item. The persisted Cradle snapshots remain UI/history records: stdout is the provider-visible result text when present, stderr is used when stdout is empty, and full stdout/stderr/exit metadata is stored under `metadata.cradle.bangResult`.
`POST /chat/sessions/:sessionId/title/regenerate` asks the active runtime to generate a fresh title from the session's first user prompt, then writes the result back to the Cradle-owned `sessions.title` field. Chat Runtime owns validation, durable binding refresh, and the explicit overwrite semantics; provider adapters own the runtime-native title-generation and best-effort provider-session rename behavior.
`POST /chat/sessions/:sessionId/rollback-last-turn` rolls back the last completed top-level user turn through the Chat Runtime provider contract, then records a `LastTurnRolledBack` fact that deletes the named message projections from Cradle's transcript. Codex implements this with app-server `thread/rollback` and `numTurns: 1`; it does not call `thread/fork` and does not revert workspace file changes. Claude Agent does not declare rollback support in this milestone.
`POST /chat/sessions/:sessionId/side-chat` creates a live-only side conversation from an existing parent session. Side chat semantics live in this module, not in provider slash-command namespaces: Chat Runtime resolves the parent runtime handle, prefers the provider `forkRuntimeSession` hook when native thread forking is available, otherwise starts a fresh live runtime session and seeds its `SideConversationRegistry` record with the parent transcript as Cradle-owned side history. It returns a `sideConversationId` for renderer-owned presentation and does not create `sessions`, `messages`, `backend_runs`, or durable provider bindings for the side conversation. `POST /chat/side-conversations/:sideConversationId/response` streams an ordinary side turn directly against the live provider handle, passes the side-local history to the runtime, appends completed side turns only to the in-memory side record, and forwards AI SDK chunks without persisting transcript rows. `DELETE /chat/side-conversations/:sideConversationId` releases the live handle; parent session archive/cleanup releases all side conversations for that parent. Once the live side registry entry expires or is released, side runtime actions fail with `side_chat_expired` instead of starting a replacement provider runtime session.
Passive observers hydrate through `GET /chat/sessions/:sessionId/messages`, then join the currently active run through `GET /chat/sessions/:sessionId/stream`; the session stream replays a coalesced active-run AI SDK chunk buffer before live chunks so late Workspace Detail joins rebuild the active assistant message through AI SDK protocol consumption without retaining every high-frequency text/reasoning/tool delta. Active-run release clears replay chunks, projection maps, pending flush timers, and final-message parts after terminal diagnostics are recorded, so closed streams cannot pin completed assistant payloads in memory.
Active streaming message snapshots are terminal-fenced against the persisted `backend_runs` row. While a run is still `streaming`, snapshot flushes may update the assistant row so renderer refresh can hydrate partial text; once a terminal fact has projected the run to `complete`, `failed`, or `aborted`, late snapshot timers, pending delta flushes, and provider finalizers stop writing and release the active run instead of changing a terminal message back to `streaming`.
`GET /chat/runs/completed` is a desktop-notification polling projection over terminal `backend_runs`. It returns `responseBody` as the completed run's latest persisted assistant response text for native notification descriptions, while `messagePreview` stays a short list-friendly preview of the same response.
Provider-native subagent/thread detail is also owned here as a session-scoped runtime capability, not as a Cradle child session. `GET /chat/sessions/:sessionId/provider-threads` lists provider threads such as Codex `subAgentThreadSpawn` entries and Claude Agent SubAgent tool-call threads, `GET /chat/sessions/:sessionId/provider-threads/:threadId` reads provider metadata, `DELETE /chat/sessions/:sessionId/provider-threads/:threadId` deletes a provider-native thread after provider-owned parent/thread validation, `GET /chat/sessions/:sessionId/provider-threads/:threadId/turns` hydrates provider turns and projected `UIMessage` records, and `GET /chat/sessions/:sessionId/provider-threads/:threadId/stream` replays live AI SDK chunks captured from the active parent run. Parent chat messages keep only the provider-owned parent tool lifecycle and reference metadata; full subagent transcripts are streamed and hydrated through provider-thread APIs instead of being persisted into parent `messages.message_json`. Chat Runtime resolves and validates the parent session/runtime/profile context, then asks Provider Runtime for an existing live side handle or durable binding; providers own native thread identity, tree membership, history projection, and notification semantics.
Background terminals are exposed as the same kind of session-scoped runtime capability. `GET /chat/sessions/:sessionId/background-terminals` lists provider-owned background terminal processes for the active provider session, and `POST /chat/sessions/:sessionId/background-terminals/:processId/terminate` asks the provider to terminate one process. Chat Runtime owns session/profile/runtime resolution and the JSON-safe HTTP contract; providers own native process identity, listing, and termination semantics.
Every backend run also writes a durable Cradle-owned run snapshot under `backend_run_snapshots` plus ordered `backend_run_snapshot_events`. This is the runtime-neutral harness envelope: Chat Runtime records lifecycle, millisecond timing, usage, terminal failure, and stable AI SDK chunk phases such as model text/reasoning and tool input/output availability. Provider-specific goal, plan, tool, MCP, context-engine, and filesystem semantics stay owned by provider adapters; snapshots only provide enough shared structure for diagnostics, replay inspection, observability export, and future automation巡检. Run snapshots are forensic records rather than session-owned transcript rows: deleting a session/run nulls those foreign keys instead of deleting the snapshot, while snapshot retention prunes completed records older than `CRADLE_CHAT_RUN_SNAPSHOT_RETENTION_DAYS` (default 30, `0` disables pruning).
Runtime goal continuation is a Chat Runtime run-lifecycle concern, with provider-owned detection and message projection exposed through `ChatRuntime.goalContinuation`. When a run ends without user cancellation while the provider snapshot still contains a continuable goal, Chat Runtime schedules a deduped follow-up system run, creates a new assistant message for that run, and asks the runtime hook to build the internal metadata-marked user message without inserting a visible user message. By default only active goals are continuable; enabling the App feature flag `continueBlockedCodexGoals` in Settings > Features also allows runtime hooks to treat blocked goals as continuable for runtimes that support that policy. User cancellation and queued user turns take precedence: cancellation remains provider-owned, and pending queue rows drain before any automatic goal continuation. `/chat/sessions/:sessionId/runtime-status` exposes `hasActiveGoal` so passive renderers can keep polling long enough to discover the new active run and join its session stream.
`GET /chat/draft-runtime-capabilities?runtimeKind=...` returns provider-owned composer capability descriptors for pre-session surfaces such as New Chat and Workspace Capsule. It is read-only, does not create or resume a backend session, and exposes only command/slot availability that can be rendered before a thread exists; live slot state such as an active Codex goal still requires a session.
`GET` / `PUT` / `DELETE /chat/composer-drafts/:surfaceId` own server-authoritative Composer drafts for renderer surfaces. The current implementation is last-write-wins over a JSON payload with soft-delete tombstones so closed surfaces and cross-window clients do not resurrect older localStorage drafts; the JSON envelope intentionally leaves room for a future Loro payload without changing the route ownership boundary.
`GET /chat/runtimes` is the runtime catalog owned by Chat Runtime. It lists builtin and plugin-registered runtime providers with display metadata, supported provider kinds, icon keys, and the surfaces that may offer the runtime (`chat`, `jarvis`). Catalog items also expose session launch ownership through `sessionLaunchMode`, so callers can distinguish provider-launched chat runtimes from agent-terminal sessions without hard-coding runtime ids. Chat, Jarvis, Automation, and future launchers should read this catalog instead of hard-coding runtime ids. Provider compatibility is registered with the runtime provider metadata, so plugin runtime ids can participate in the same provider-target validation path as Codex, ACP Chat / Cloud Agent SDK, Claude Agent, Standard, and HiJarvis.
`GET /chat/runtimes/health` runs optional provider health checks for already-registered runtimes. Providers that do not expose `healthCheck` return `unknown`; health probing is intentionally separate from catalog listing so ordinary runtime selectors do not trigger provider IO.
`GET /chat/sessions/:sessionId/capabilities` owns the session-scoped runtime capability inventory returned to the renderer. Its payload now includes runtime-native slash commands, provider-projected UI slots, and skill ids, so composer presentation can stay provider-driven instead of hard-coded in the frontend. Runtime UI slots may also declare `commandAction` and `requiresSession`, allowing providers to define whether a slash row inserts text, submits a command, opens a Cradle UI action, or is hidden from pre-session draft composers without frontend runtime-id branches.
Internally, `ChatRuntimeCapabilities` is no longer this renderer presentation payload. It is the static provider routing contract declared on each runtime at construction time: live steering, shell execution, runtime settings updates, UI slot state support, dynamic provider capability discovery, and session model-switch semantics. Provider-owned renderer descriptors are represented separately as runtime presentation capabilities and are returned through `getPresentation` / `getDraftPresentation`, preserving the existing HTTP payload while keeping routing flags cheap and synchronous.
`GET /chat/sessions/:sessionId/ui-slot-states` returns provider-owned state for declared UI slots. Renderer surfaces are selected from `RuntimeUiSlot.surfaces`: `composerState` can render primary workflow state near the composer, `slashCommand` can enter the slash panel, `toolbarPicker` can expose option summaries next to existing toolbar controls, `runtimePanel` can render Right Aside Runtime summaries, and `streamEvidence` is consumed by message/tool blocks from provider-emitted chunks. Polled slot state is a current snapshot and must not be used to synthesize transcript-positioned message-bubble content. Codex `goal` is read from the app-server thread goal object; Codex `compact` separates context-compaction item lifecycle from thread token-usage meter state, using `contextCompaction` item start/complete as the primary lifecycle source and deprecated `thread/compacted` only as a compatibility signal; Codex `status`, `model`, and `reasoning` are projected from thread start/resume responses, settings/status notifications, config, model list, and model-provider capability reads. Codex `plan`, `toolActivity`, `mcp`, `diff`, `terminal`, `approvals`, `alerts`, `filesystem`, and `search` are projected from their app-server notifications and bounded provider snapshots; Codex `terminal` also reads provider-owned background terminal rows at render time so Composer can show process ids, CPU, memory, and terminate actions without knowing Codex JSON-RPC method names. Claude Agent `progress` is projected from the latest normalized TodoWrite state or structured TaskCreate/TaskUpdate/TaskList state in the Claude Agent provider snapshot; the adapter does not parse human-facing task result strings to recover IDs. Codex `usage` is projected from `account/rateLimits/read` and rate-limit notifications, including primary/secondary window duration and reset timestamps so renderer composer and runtime-panel surfaces can show 5h/weekly-style account usage without reading Codex protocol payloads directly; `skills`, `plugin`, `crew`, and `config` use provider list/read methods at render time. Codex `crew` enumerates subagent threads with paginated provider `thread/list` and uses recent parent-turn reads only for call lifecycle summaries. Chat Runtime stores only the provider snapshot needed to replay those projections; it does not own goal, compaction, plan, progress, tool, MCP, model, reasoning, diff, terminal, approval, alert, filesystem, search, usage, skill, plugin, crew, or config lifecycle semantics.
`GET /chat/sessions/:sessionId/context-usage` exposes provider-owned context window usage details when the active runtime supports them. The response preserves the runtime-neutral `RuntimeContextUsage` sections/items shape so renderers can show system prompt, message, tool, file, skill, and provider-specific usage groups without inventing a Chat-owned tokenizer projection. Runtimes that only expose aggregate context usage should continue to publish `compact` UI slot state; callers must treat a `null` `usage` payload as "details unavailable" rather than as zero tokens.
Quick question `/btw` support is declared by providers as a `quick-question` UI slot with `slashCommand` and `composerState` surfaces, while `POST /chat/sessions/:sessionId/quick-question` owns the stateless no-history stream. Providers own whether the slot is present; Chat Runtime only transports the declaration and dispatches the quick-question hook.
User attachments are represented as AI SDK `FileUIPart` entries in `UIMessage.parts`; the OpenAI-compatible AI SDK provider converts those through `convertToModelMessages`, Codex maps image parts into app-server `UserInput` items, and text-only runtime providers reject unsupported parts explicitly instead of silently dropping attachments. Chat-owned context parts such as `data-cradle-skill` are persisted in the same snapshots but projected only by providers that support them.
`session_events` is the canonical chat-runtime fact log for session-owned run, message, queue, steer, provider-title, and runtime interaction lifecycle transitions such as tool approval and user input requests. `messages`, `backend_runs`, `chat_session_queue_items`, durable run snapshots, and runtime-owned `sessions` fields are same-transaction projections of those facts for HTTP reads, observability, notifications, and renderer hydration; interaction lifecycle facts are tail-only synchronization facts and do not write extra projection tables. Session metadata creation, archive, read/unread, and deletion remain owned by the session module. `POST /chat/sessions/:sessionId/cancel` owns the canonical aborted transition: it marks active runs/messages terminal before asking the provider to stop, ignores late provider chunks after terminal state, and projects persisted `streaming` rows as `aborted` only for explicit user cancellation. Server startup and message hydration treat orphaned persisted `streaming` runs as process interruptions, not user cancellation, and append a deterministic `RunFailed` fact with `response.interrupted`; terminal fact projection keeps `messages`, `chat_session_queue_items`, and `backend_run_snapshots` from staying in streaming/running states after crash or interrupted finalization. Recovery also repairs historical terminal projection drift from `session_events`: if a terminal run fact exists but the assistant message row is still `streaming`, it reapplies the matching `AssistantMessageCompleted` fact, and if a durable run snapshot contradicts the terminal run fact, it reconciles the snapshot terminal fields to that fact. Ordinary read APIs do not append repair events. `GET` / `PATCH /chat/sessions/:sessionId/runtime-settings` reads and updates Cradle-owned `accessMode` / `interactionMode` controls plus the direct Chat Session Claude Agent model matrix override, persists access/interaction under `sessions.config_json.runtimeSettings` and the matrix under `sessions.config_json.claudeAgent.modelAliases`, and forwards live access/interaction updates only to providers that implement the runtime settings side-channel.
Runtime skill path discovery is cached per workspace for a short window so simultaneous chat turns do not repeatedly perform synchronous directory scans during provider startup. Providers still receive Cradle-owned skill paths only as read inputs; skill lifecycle stays under the skills namespace.
Runtime provider inputs carry `sessions.agentId` alongside `workspacePath` when a session is agent-scoped. Chat Runtime owns resolving that database context and forwarding it; provider directories own any runtime-specific filesystem interpretation such as Codex using the agent home as cwd.
Provider-native session titles are projected through the `StreamTurnInput.reportSessionTitle` callback. Chat Runtime owns the write into `sessions.title`; providers only report titles they read from their own runtime/session metadata. Explicit title regeneration uses the provider `generateSessionTitle` hook and intentionally overwrites a user-edited title only for that user-triggered action.
Route metadata includes `x-cradle-cli` descriptors for non-streaming generated CLI commands.
Chat stream traces are opt-in through `CRADLE_CHAT_STREAM_TRACE=1` and are written to `CRADLE_DATA_DIR/chat-runtime/traces/*.jsonl` with bounded payloads by default. Use `CRADLE_CHAT_STREAM_TRACE_FULL=1` only for short diagnostic runs because full provider raw events, mapper output, runtime chunks, and emitted AI SDK SSE frames can be very large.
Lightweight runtime profiling is opt-in through `CRADLE_CHAT_RUNTIME_PROFILE=1`; it logs per-run stream/finalize timings, final snapshot bytes, replay chunk count, output diagnostics, and process memory deltas without writing extra trace files.
Chat Session follow-up ownership is split by transport semantics. `/chat/sessions/:sessionId/queue` stores only durable follow-up turns in `chat_session_queue_items`, including per-item runtime settings snapshots and selected context parts for compatible runtimes, then drains those rows after the active run completes. `/chat/sessions/:sessionId/steer` is live same-turn guidance: it requires an active provider run with the optional `steerTurn` hook, persists the accepted visible user message with `metadata.cradle.continuation.mode = "steer"`, and fails explicitly when the active run cannot accept guidance. Steer is never written to the durable queue, never reordered with queue rows, and never silently replayed as a later turn.
Codex app-server HTTP routes are owned here only as session-scoped Chat Runtime capabilities. `/chat/sessions/:sessionId/codex/app-server/capabilities` resolves the session and returns the provider-native capability manifest, `/invoke` performs one-shot provider-native calls, and `/stream` opens a provider-native notification stream. Chat Runtime owns the Cradle session/profile/runtime context resolution and persists the adapter-updated provider snapshot after an invoke. The Codex adapter owns the generated app-server method surface, JSON-RPC bridge execution, server-request/tool mapping, and provider-native state projection such as goal invoke results.
Provider tool envelope shape is owned by `../chat-runtime-providers/tools`, while provider-specific tool identity and payload semantics live in each provider directory such as `../chat-runtime-providers/claude-agent/tools` and `../chat-runtime-providers/codex/tools`. Provider adapters emit tool input/output payloads as `{ identifier, apiName, args, result }` envelopes while Chat Runtime only persists and streams the AI SDK chunks. Claude Agent `TodoWrite` state is carried inside the Claude Code tool result envelope as `result.pluginState.todos`, and the existing terminal tool message snapshot persistence writes that state into `messages.message_json`; the Claude Agent provider also stores the latest normalized TodoWrite list in its provider snapshot for the runtime-neutral `progress` UI slot.

## Files

- `index.ts`: Thin Elysia route aggregator for `http/*.routes.ts`.

- `model.ts`: Public `ChatRuntimeModel` schema aggregate used by the route modules.
- `model/common-schemas.ts`: Shared HTTP schemas for messages, queue items, run traces/snapshots, runtime catalog/model descriptors, provider threads, session-tail events, and request parts.
- `model/ui-slot-schemas.ts`: Runtime UI slot declaration/state schemas and provider-owned context-usage schemas.
- `runtime.ts`: Composition root for chat-runtime public APIs, dependency wiring, active-run streams, queue drain, side chat, cancellation, and recovery entry points.
- `bang-command.ts`: standard-runtime single-line Composer bang command executor; resolves the session workspace, spawns the command through the host shell with bounded stdout/stderr capture, handles timeout/request abort, and persists command/result user messages with Cradle-owned bang metadata.
- `pending-tool-approval.ts`: in-memory active-run registry for provider-native tool approval requests; resolves user decisions from the HTTP route back into the provider request promise and rejects pending approvals when a run ends.
- `es/event-store.ts`: append-only `session_events` persistence, aggregate version assignment, and stored event hydration.
- `es/event-tail.ts`: Slim session/global event-tail DTO reads and in-process SSE subscriber fanout.
- `es/events.ts`: chat session event payload contracts that reuse existing DB row shapes and AI SDK `UIMessage` snapshots.
- `es/aggregate.ts`: pure in-memory replay of session events for active-run detection and deterministic interrupted-run finalization.
- `es/projectors.ts`: synchronous same-transaction projections from session events into `sessions`, `messages`, `backend_runs`, `chat_session_queue_items`, and `backend_run_snapshots`.
- `es/session-actor.ts`: per-session serialized command execution so one session writes monotonically versioned events without concurrent SQLite writes.
- `es/commands.ts`: command helpers that append chat session events and project them, including queue transitions, terminal run fact helpers, and last-turn rollback facts.
- `es/recovery.ts`: interrupted-run finalization, missing terminal fact projection, and terminal projection drift repair.
- `ui-message.ts`: trusted internal `UIMessage` helpers for stored message-JSON parsing, assistant/user message construction with text, context, and file parts, Codex internal goal-continuation metadata, bang command/result metadata, and text extraction. This file intentionally avoids Cradle-owned Zod schemas over AI SDK message internals. (Renamed from `message-snapshots.ts`; unrelated to the durable run snapshots in `run-snapshot.ts`.)

- `pending-user-input.ts`: in-memory pending user input registry keyed by session and provider request id; owns request creation, duplicate/stale validation, run-scoped rejection, answer resolution, summary projection for live status/desktop notifications, and synthetic resolved tool-output publication for active run streams.
- `composer-drafts.ts`: server-authoritative Composer draft persistence with LWW revision increments, JSON payload storage, and soft-delete tombstones keyed by renderer `surfaceId`.
- `ui-message-input.ts`: provider-boundary helpers for extracting text from `UIMessage` and rejecting attachments for text-only runtimes.
- `context/turn-context.ts`: system prompt, Chronicle long-term memory injection, and bounded `CradleTurnTranscript` history resolution for ordinary turns.
- `provider-threads/live-streams.ts`: provider-native thread live chunk replay/fanout state, replay buffer limits, provider-turn start chunk synthesis, and subscriber cleanup.
- `run/final-message-projection.ts`: incremental `UIMessageChunk` to final assistant `UIMessage` projection, including text/reasoning flushing and tool input/output/approval part mutation.

- `run/turn-draft.ts`: event-sourced user/assistant draft turn creation from text/files/context or supplied AI SDK messages.
- `run/terminal-finalizer.ts`: terminal chunk publication, terminal message snapshot commit, run settlement, and terminal binding persistence.
- `run/active-run-snapshot.ts`: active-run snapshot start/event/finalization wrapper around durable run snapshots.
- `run/active-run-release.ts`: active-run memory/resource cleanup and stale persisted terminal fence release.
- `run/runtime-goal-continuation.ts`: Runtime goal continuation detection through provider hooks, backoff, timer scheduling, and dependency-injected continuation run startup.
- `run/snapshot-events.ts`: runtime-neutral run snapshot phase mapping, chunk payload summarization, event append, and terminal snapshot finalization.
- `run/output-diagnostics.ts`: provider output event counters and empty-output terminal validation for streamed turns.
- `run/profile.ts`: opt-in chat runtime profiling session that records stream/finalize timings, memory deltas, replay/final-message sizes, and output diagnostics.
- `run/usage.ts`: total and per-step token usage persistence plus model-cost estimation used by run finalization and snapshot events.
- `queue/session-queue.ts`: durable queue DTO/input types, queue row JSON parsing/serialization, persisted thinking-effort normalization, runtime-settings projection, and queue row ordering.
- `queue/drain.ts`: durable queue drain scheduling, event-sourced queue item claiming/release/failure transitions, and dependency-injected queued run startup.

- `side-chat/create.ts`: live-only side conversation creation, parent runtime handle resolution, provider fork/start dispatch, host lease registration, and side conversation registry seeding.
- `side-chat/live-stream.ts`: live-only side conversation provider stream transport, final assistant projection, completion history callback, and best-effort cancellation.
- `chat-runtime-provider-registry.ts`: runtime provider registry for ACP Chat, OpenAI-compatible, Claude Agent, Codex, System Agent (`jar-core`), plugin runtimes, and debug/mock variants. It creates the shared `ProviderContext`, validates runtime contract shape and capability-hook consistency at registration time, registers builtin adapters through provider-owned factory functions, publishes runtime catalog metadata and optional health status, updates provider-kind compatibility, allows builtin-owned runtime replacement for tests while preserving plugin namespace protection, and keeps cached runtime skill path discovery under Chat Runtime because skills are resolved per chat workspace.
- `runtime-settings.ts`: JSON helpers for reading and writing session runtime settings and Claude Agent session matrix overrides without creating a separate session settings table.
- `runtime-settings-api.ts`: HTTP-facing service for `GET` / `PATCH /chat/sessions/:sessionId/runtime-settings`; merges access/interaction patches, writes optional Claude Agent matrix overrides, and attempts live access/interaction updates for active runtimes.
- `runtime-settings-model.ts`: TypeBox schemas for runtime settings and Claude Agent session matrix request/response payloads.
- `runtime-provider-types.ts`: chat runtime provider contracts, including static provider routing capabilities, runtime presentation descriptors, shared provider dependency context, structured provider error tags, runtime catalog metadata, UI slot surfaces, provider-owned UI slot state unions for goal, compact usage and lifecycle, plan, tool activity, MCP, model, reasoning, status, diff, terminal, approvals, alerts, filesystem, skills, plugin, search, crew, usage with account rate-limit windows, and config; provider-native thread list/read/turns/live-event and app-server hooks; optional `forkRuntimeSession`, `steerTurn`, title generation, shell execution, runtime settings update, UI slot state, dynamic capability, health-check, and dispose hooks; and the `reportSessionTitle` callback for provider-native title projection.

`../chat-runtime-engine` owns provider-agnostic AI SDK execution helpers, compaction, and provider selection. `../chat-runtime-providers` owns concrete backend adapters. Chat Runtime depends on both through narrow contracts and keeps HTTP routes, persistence, queueing, trace files, and run lifecycle semantics here.

## Provider MCP Ownership

The server MCP registry owns MCP server discovery for both plugin-registered servers and host-owned builtin servers such as Chronicle. Chat runtime providers in `../chat-runtime-providers` only read that registry when their upstream runtime supports MCP-style tool servers:

- Claude Agent receives the registry through SDK `mcpServers` query options.
- ACP receives the registry on `newSession`, `loadSession`, and `unstable_resumeSession` as ACP `McpServer[]`.
- Codex receives the registry through `config.mcp_servers`.

OpenAI-compatible and System Agent (`jar-core`) providers do not currently expose a compatible plugin-MCP injection surface in this module.

## Live Steer Ownership

`ChatRuntime.steerTurn` is optional and live-only. The `/steer` route calls it only for the currently active run after validating that an explicitly requested provider target still points at that run's provider target. Composer model, thinking effort, and runtime settings describe future turns and durable queue items; they do not define live steer identity for an already-running provider turn. If the provider accepts the message, Chat Runtime inserts a completed visible user message with `metadata.cradle.continuation.mode = "steer"` for renderer presentation. If there is no active run, the provider lacks the hook, the active provider target does not match, the provider rejects the input, or message persistence fails, the route returns an explicit error. It does not create or mutate a `chat_session_queue_items` row, so live guidance cannot be cancelled, reordered, drained, or replayed as a later durable queue turn.

Claude Agent supports this hook through the Claude Agent SDK streaming-input query plus `interrupt()`. Codex supports this hook through the Codex app-server JSON-RPC protocol: the provider keeps the active app-server client, `threadId`, and `turnId`, then sends `turn/steer` with the expected active turn id. ACP, OpenAI-compatible, System Agent, and mock runtimes reject `/steer` unless their providers add a real side-channel later; callers that want deferred execution must submit a separate `/queue` request.

Chronicle registers a builtin `chronicle` MCP server from `../chronicle/mcp.ts`, exposing read-oriented memory, activity segment, and knowledge-card tools to the providers above. Chat Runtime remains a consumer of that registry and does not own Chronicle semantics or writes.

## Chronicle Context

Chat runtime reads Chronicle-owned memory context through `../chronicle/agent-context.ts` during turn context resolution. It uses the current user message as a keyword query, injects a small read-only `Chronicle long-term memory context` block into the system prompt when relevant memory or knowledge exists, and redacts common sensitive values before the prompt reaches the selected runtime provider. The helper intentionally avoids importing the main Chronicle service so chat execution does not create a Chronicle-to-chat dependency cycle or trigger local embedding health probes during ordinary turns.

## ACP Client Filesystem Ownership

ACP `fs.writeTextFile` writes to the client filesystem, not to Cradle-owned data. The legacy Cradle approval SSE owner has been removed from chat runtime, so `../chat-runtime-providers/acp/connection-manager.ts` must fail closed and leave the target file unchanged until ACP filesystem writes are represented as AI SDK tools with native `needsApproval`. Do not reintroduce a separate approval stream or wrapper for this path.
